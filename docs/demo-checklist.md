# Demo Checklist

## 使用方式

这份清单用于明天收尾时快速自检和演示，不讲实现细节，只关注“能不能顺利演示”。

建议使用顺序：

1. 先完成云端更新与重启
2. 再做 5 分钟自测
3. 最后按本清单对外演示

---

## 一、演示前 5 分钟自检

### 云端服务

- 执行 `./scripts/appctl.sh update --no-cache` 或 `./scripts/appctl.sh restart`
- 执行 `./scripts/appctl.sh status`
- 执行 `./scripts/appctl.sh logs backend`
- 执行 `./scripts/appctl.sh logs frontend`

确认点：

- 前后端容器都在运行
- 后端无启动报错
- 前端无构建失败

### 浏览器入口

- 打开 `http://115.191.43.252:18761`
- 确认登录页或首页能正常打开
- 确认没有明显白屏、接口 500、CORS 报错

### 预置数据

- 至少有 1 条 GPU 资源可展示
- 至少 1 家真实 Provider 或明确的 bind/mock 兜底资源可展示
- 测试页面不会因为空数据直接报错

---

## 二、正式演示流程

### 1. 首页 / 资源市场

演示动作：

- 进入 GPU 市场页
- 展示左侧筛选栏
- 按型号、区域、价格做 1 到 2 次筛选
- 点开一个目标资源

需要讲清楚：

- 页面支持统一展示多 Provider 资源
- 当前 UI 已具备资源筛选、配置、一键开机的完整入口

通过标准：

- GPU 列表正常显示
- 筛选后结果会变化
- 至少有 1 条资源能进入配置页

### 2. 配置页 / 一键开机

演示动作：

- 进入配置页
- 展示模板、磁盘、时长、预算摘要
- 点击“一键开机”

需要讲清楚：

- 当前页面提供模板化开机
- 会做预算估算
- 创建请求通过统一 Provider 抽象发起

通过标准：

- 能成功发起实例创建
- 页面能跳转或出现实例创建结果

### 3. 实例列表

演示动作：

- 打开实例列表页
- 展示新实例的状态

需要讲清楚：

- 实例列表与用户隔离绑定
- 当前实例记录包含 provider、状态、创建时间等摘要信息

通过标准：

- 列表中出现刚创建的实例
- 状态展示正常

### 4. 实例详情 / Dashboard

演示动作：

- 进入实例详情页
- 展示顶部状态条
- 展示 Telemetry 面板
- 展示 Connect / Tests / 外网可达性相关信息

需要讲清楚：

- dashboard 已具备监控控制台结构
- 数据来源优先真实，必要时有 fallback
- 状态由 heartbeat 驱动，而不是纯前端写死

通过标准：

- 页面能正常加载
- 至少展示 CPU / 内存 / GPU / 磁盘中的主要指标
- Connect 面板能展示至少 SSH 或 provider_instance_id 信息

### 5. 性能测试

演示动作：

- 点击“运行性能测试”
- 等待结果返回
- 展示测试结果

需要讲清楚：

- 性能测试和连通性测试已拆分
- 结果可存储、查看、导出

通过标准：

- 性能测试可触发
- 页面出现结果或测试记录

### 6. 外网可达性测试

演示动作：

- 点击“运行连通性测试”
- 展示返回结果

需要讲清楚：

- 连通性测试独立于性能测试
- 用于说明实例对外访问可用性

通过标准：

- 连通性测试能触发
- 页面有返回结果或状态记录

### 7. 导出结果

演示动作：

- 导出测试结果
- 展示导出的文件或下载行为

需要讲清楚：

- 支持结构化导出
- 便于测试归档与复盘

通过标准：

- 文件能成功下载
- 导出内容不是空文件

### 8. 销毁实例

演示动作：

- 返回实例页或实例列表
- 触发删除 / 销毁

需要讲清楚：

- 销毁会走 Provider 抽象
- 平台负责统一管理不同 Provider 生命周期

通过标准：

- 实例成功销毁
- 状态或列表有变化

## 三、演示时推荐话术

### 如果真实 Provider 已接通

- “当前至少 1 家供应商已经实现真实调度模式，支持统一资源展示、创建和销毁。”

### 如果只有 1 家真实、其余是 bind/mock

- “当前 1 家供应商已经完成真实 API 调度，其余供应商先通过 bind/mock 兜底，但架构层已经统一到 Provider 抽象，后续只需补实现。”

### 如果被问到为什么不继续改 UI

- “当前 UI 已经完成控制台式骨架，后续重点放在真实 Provider、Agent、测试和部署链路，而不是继续做视觉微调。”

---

## 四、失败时的应急顺序

### 首页打不开

- 先看 `./scripts/appctl.sh logs frontend`
- 再看 `./scripts/appctl.sh logs backend`

### 接口报错

- 先确认后端是否启动
- 再确认 CORS 与 API 地址

### 真实 Provider 创建失败

- 快速切回 bind/mock 演示资源
- 明确说明真实调度正在对接，当前架构已完成抽象接入

### 指标没有刷新

- 检查 heartbeat 是否到达
- 检查 `agent_service` 与 dashboard fallback

## 五、最终收工判断

满足以下条件即可认为明天可以收工：

- 市场页可展示并筛选资源
- 至少 1 次成功创建实例
- 实例详情页能展示 dashboard
- 性能测试可运行
- 外网可达性测试可运行
- 导出可用
- 销毁可用
- 云端可通过脚本完成更新和重启
